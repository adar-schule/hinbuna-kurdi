One Pick at Registration, Two Independent Settings

When a user registers (student or teacher), they pick their language. This single pick sets both ui_locale and content_locale to the same value initially. However, users can change them independently later in Settings — for example, keeping the UI in German while viewing lesson translations in English.

Hybrid Translation Storage (Pattern C + D)

Translation storage uses a hybrid approach decided in Phase 2:

  • Pattern C — Translation tables for flat fields (names, descriptions) on courses, modules, units, and lessons. Each parent table has a corresponding *_translations table with columns for locale, translated text, translation_source, source_locale, translated_by, and reviewed_at.
  • Pattern D — JSONB with locale keys for activity content only. Activities have deeply nested, type-specific content where translation tables would be impractical. Locale keys ("de", "en", etc.) are embedded directly in the JSON fields.

See Section 6 below for the full *_translations table definitions. The JSONB examples below apply only to activities:

// Activity content uses JSONB with locale keys (Pattern D) { "translation": { "de": "Guten Tag / Hallo", "en": "Good day / Hello", "tr": "İyi günler / Merhaba" } }

What Does NOT Get Translated

  • Kurdish content — word, sentence, ku fields are the teaching language
  • Structural fields — id, slug, order_index, type

  • Audio/Image URLs — Language-agnostic resources

How Teachers Create Multi-Lang Content

Teachers must handle the fact that every translatable field needs multiple languages. The teacher screen (T5 Activity Editor) shows per-language input fields with color-coded chips.

1
Teacher writes Kurdish content
The teaching language — word, sentence, Kurdish text. This is always manual, never AI-generated.
2
Teacher writes German translation
DE is the primary support language. Mamoste writes this manually — it's the quality source for all other translations.
3
AI translates DE → other languages
From the German text, AI generates EN, SV, DA, IT, TR, FA, AR translations. Not from Kurdish — AI Kurdish quality is unreliable.
4
Teacher reviews & adjusts
AI translations are pre-filled but editable. Teacher can correct any language they know. All languages visible in one form.
Why not translate from Kurdish? Current AI models (GPT, Claude, etc.) produce unreliable Kurdish translations — especially Kurmanji dialect. German → other languages is high quality. As Kurdish AI improves, we can add KU as a secondary source.

Fallback Chain

When a translation is missing for the user's locale, the t() helper falls through this chain:

function t(val) { if (!val) return ''; if (typeof val === 'string') return val; // Legacy single-string var locale = getLocale(); // DB or localStorage return val[locale] // 1. User's language || val['de'] // 2. German (primary support) || val['en'] // 3. English || Object.values(val)[0] // 4. First available || ''; // 5. Empty string }
DE before EN in fallback. Since DE is always manually written by Mamoste, it's the most reliable translation. EN may be AI-generated. If both are missing, take whatever's available.

Architecture Decision: Hybrid C+D

  • Pattern C for flat fields — Separate *_translations tables for courses, modules, units, lessons. Queryable, indexable, industry standard (Moodle, Canvas, Strapi).
  • Pattern D for activities — Inline JSONB for nested, variable content (MCQ options, gap fill blanks, etc.). Structure varies per activity type.
  • Translation metadata — Every translation row tracks source (human/machine/machine_edited), who translated, when reviewed. Enables teacher review workflows.
  • Whitelabel-safe — supported_locales table per deployment. Add/remove languages without schema changes.
